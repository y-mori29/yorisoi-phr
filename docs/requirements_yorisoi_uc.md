# よりそい カスタマイズ要件定義書
## 潰瘍性大腸炎（UC）患者向け機能拡張

---

## 1. ドキュメント概要

| 項目 | 内容 |
|------|------|
| 対象サービス | よりそい（AI医療相談・診察サポートサービス） |
| 対象疾患 | 潰瘍性大腸炎（Ulcerative Colitis / UC） |
| 対象ユーザー | 治療が安定している寛解期の患者 |
| ドキュメント種別 | エンジニア向け要件定義書（初版） |
| 作成日 | 2026年4月 |

---

## 2. 背景・調査概要

### 2-1. ユーザー属性

インタビュー対象は、潰瘍性大腸炎と診断済みで、現在は治療が安定している寛解期の患者。
急性期・入院中ではなく、日常生活を送りながら定期的に通院・服薬を継続しているフェーズ。

### 2-2. 調査から得られた主要インサイト

- **記録が続かない根本原因は「出口がないこと」**  
  日常的に記録を試みた患者はほぼおらず、記録しても断片的になり振り返らない。  
  ドクターに見せる・活用できる場面があれば継続動機になる可能性がある。

- **「診察室で手早く伝えられない」という強いペイン**  
  ドクターにはちゃんと伝えたいが、限られた診察時間の中で口頭で伝えるのが難しい。  
  記録・情報整理のアウトプット先は「診察前の共有」に設定するのが適切。

- **「自分の経歴を時系列で俯瞰したい」というニーズ**  
  新しい医療機関を受診するたびにゼロから説明しなければならないことへの疲弊感が強い。  
  患者自身が「自分の年表」として経歴を把握・提示できる手段を求めている。

## 3. ユーザーペインと要求機能の対応

| # | ペイン | 要求機能 | 優先度 |
|---|--------|----------|--------|
| A | 薬が変わったとき、薬名・変更理由・副作用を忘れる | 薬情報管理機能 | 高 |
| B | 排便回数・便の様子を日常的に記録したいが続かない | 症状日次記録機能 | 中 |
| C | 新しい医療機関でゼロから説明しなければならない | 患者年表（タイムライン）機能 | 最高 |
| D | 外出中に常にトイレを探している | トイレ探しアシスト機能 | 低（既存技術資産により実装難易度は低〜中） |

## 4. 機能要件詳細

---

### 機能C：患者タイムライン（Patient Timeline）

#### 背景・目的

潰瘍性大腸炎患者は、病院を変える場面（引っ越し・紹介・救急・セカンドオピニオン）で、  
これまでの治療歴を口頭でゼロから説明しなければならないという強いストレスを抱えている。  
患者自身が「自分の経歴の年表」を持ち、必要な場面で提示・共有できる機能を提供する。

#### 含めるべき情報カテゴリ

- 診断日・診断名
- 入院歴（期間・理由）
- 薬の変更履歴（薬名・変更理由・変更日）
- 主な検査歴（内視鏡など）
- 主治医からの指示・治療方針の変更
- 現在の症状の状態（寛解・再燃など）

#### 要件

- イベントを時系列順に表示する（タイムライン形式）
- 表示粒度を「大まかな年単位」から「日単位」まで自由に切り替えられること
- イベントの追加・編集・削除ができること
- 診察前に医師へ見せる用の「サマリー表示モード」を持つこと
- 将来的に、よりそいの診察記録（音声要約）と自動連携できるよう設計すること（Phase 2想定）

#### 想定ユースケース

1. 患者が新しい病院を受診する前日、タイムラインを開いて過去の治療歴を確認する
2. 診察室でスマートフォンのタイムライン画面を医師に見せる
3. 救急受診時に過去の薬歴・治療歴を素早く提示する

### 機能A：薬情報管理（Medication Manager）

#### 背景・目的

潰瘍性大腸炎の治療では、薬が段階的に変更されることが多い。  
患者は処方変更時に医師から説明を受けるが、内容を忘れやすく、  
後になって「なぜ変わったのか」「どんな副作用があるのか」がわからなくなるケースが多い。

#### 具体的なペイン

- 服薬タイミング（食後・食前など）の混乱よりも以下が主な問題
- 副作用の注意点を忘れる
- 薬を変えた理由がわからなくなる
- 薬の名前自体が覚えられない
- 複数の薬を使っていてどれがどれかわからなくなる

#### 要件

- 処方されている薬を一覧管理できること
- 上記薬剤マスタからプリセット選択できること（薬名が覚えられないペインへの対応）
- 各薬に「変更日・変更理由・副作用メモ・医師のコメント」を紐づけられること
- 薬の変更履歴を時系列で確認できること（機能Cのタイムラインと連携する）
- よりそいの診察要約から薬情報を自動抽出・登録する仕組みを検討すること（Phase 2想定）

### 機能B：症状日次記録（Daily Symptom Log）

#### 背景・目的

UC患者は毎日の排便状態を把握することが病状管理に重要だが、  
現状は記録を行っている患者がほぼいない。  
記録が続かない理由は「入力コストが高い」「記録しても使い道がない」の2点に集約される。  
入力コストを極限まで下げ、かつ記録した内容が診察で活用される出口を作ることが継続の鍵。

#### 記録する主な内容

- 排便の回数（1日あたり）
- 便の性状（Bristol Scale または独自の簡易スケール）
- 血便の有無
- 腹痛・体調の主観スコア（任意）

#### 要件

- **音声入力**で記録できること（例：「今日3回、普通」と話すだけで登録）
- **画面タップ**で記録できること（ワンタップ〜2タップで完結する設計）
- 記録は1日単位で集計・表示されること
- 記録の履歴をグラフ・カレンダー形式で振り返れること
- 診察前に「直近の記録サマリー」を出力し、医師に見せられる形式にできること

### 機能D：トイレアシスト（Restroom Finder）

#### 背景・目的

UC患者は外出時に常にトイレの場所を意識している。  
緊急性が高いケースもあるため、素早く近くのトイレを見つける手段が求められる。

#### 要件

- LINEのよりそいチャット内から「近くのトイレ」を検索・表示できること
- LIFF（LINE内ブラウザ）上に地図を表示し、LINEを離れずに完結すること
- 現在地を取得して半径500m程度のトイレを地図上にピン表示すること
- 実装コストは低〜中程度と見込む（後述の技術資産活用により）

#### 技術方針

**採用構成：Google Maps JavaScript API + Places API（Nearby Search）をLIFF上で実装**

| レイヤー | 採用技術 | 備考 |
|----------|----------|------|
| 地図表示 | Google Maps JavaScript API | hybridモード推奨 |
| 周辺検索 | Places API（Nearby Search） | キーワード：`トイレ` / `公衆トイレ` |
| 現在地取得 | `navigator.geolocation.watchPosition()` | liff.getLocation()よりリアルタイム性が高い |
| 動作環境 | LIFF（LINE内ブラウザ） | liff.init()からの標準起動フロー |

**料金について**  
Google Maps Places API は月$200の無料枠があるため、初期フェーズは実質無料で運用可能。
ユーザー数拡大後はMapboxへの移行を検討する。

#### 実装の核となるコード構造

```javascript
// 1. LIFF初期化 → 現在地取得
liff.init({ liffId: 'YOUR_LIFF_ID' }).then(() => {
  navigator.geolocation.watchPosition(pos => {
    const ll = { lat: pos.coords.latitude, lng: pos.coords.longitude };
    initMapWithLocation(ll);
  }, err => { alert('GPS取得失敗: ' + err.message); }, { enableHighAccuracy: true });
});

// 2. 地図初期化 → Places APIで周辺トイレ検索
function initMapWithLocation(ll) {
  const map = new google.maps.Map(document.getElementById('map'), {
    center: ll, zoom: 16, mapTypeId: 'hybrid'
  });
  const service = new google.maps.places.PlacesService(map);
  service.nearbySearch({
    location: ll, radius: 500, keyword: 'トイレ'
  }, (results) => {
    results.forEach(place => {
      new google.maps.Marker({ position: place.geometry.location, map, title: place.name });
    });
  });
}
```

#### 技術資産の活用について

既存のゴルフアプリ（golf_shot_map）が Google Maps JavaScript API + `navigator.geolocation.watchPosition()` を本番環境で稼働済みのため、以下のコードが直接転用可能。

- **リポジトリ：** https://github.com/Keeeeei-Soeda/golf_shot_map
- **参照ファイル：** `app.js`（GPS・マーカー処理）、`index.html`（Maps API初期化）

- GPS取得・マーカー表示のロジック全般
- Google Maps の初期化コールバックパターン（`window.initMap`）
- マーカーアイコンのSVGカスタマイズ手法

差分として新たに実装が必要なのは **Places APIの呼び出し部分のみ**であり、実装難易度は当初想定より低い。

#### 想定ユースケース

1. 患者が外出中によりそいを開き「近くのトイレ」とメッセージを送る
2. LIFFが起動し、現在地周辺のトイレがマップ上にピン表示される
3. 任意のピンをタップして詳細（施設名・距離）を確認する

## 5. データ設計・インフラ方針

### 6-1. 現行よりそいのアーキテクチャ

現行のよりそいには**RDB（リレーショナルDB）が存在しない**。
データはすべてGoogle Cloud Storage（GCS）にJSONファイルとして保存されており、バックエンドはCloud Run（Node.js 18 + Express）で稼働している。

```
現行の主要インフラ
├── フロントエンド：ロリポップサーバー（LIFF / index.html）
├── バックエンドAPI：Google Cloud Run（Node.js 18 + Express）
├── ストレージ：Google Cloud Storage（音声・要約・メタデータ）
├── AI：Google Cloud Speech-to-Text + Gemini
└── 配信：LINE Messaging API
```

患者の識別は **LINE UserID** で行っており、既存の認証基盤として活用可能。

### 6-2. 新規DB追加の必要性

今回追加する機能A〜Cは、GCSのJSONファイル管理では賄えないデータ特性を持つ。

| 機能 | データ特性 | GCS JSONで賄えるか |
|------|-----------|-----------------|
| 患者年表（機能C） | ユーザーによる追加・編集・削除・時系列ソート | ❌ |
| 症状日次記録（機能B） | 毎日蓄積・日付集計・グラフ表示 | ❌ |
| 薬情報管理（機能A） | 変更履歴・カテゴリ紐付け・複数薬の管理 | ❌ |

→ 新たに**Cloud Firestore（Firebase）**をデータストアとして追加する。

### 6-3. Cloud Firestoreを採用する理由

| 理由 | 内容 |
|------|------|
| GCPで統一できる | Cloud Run・GCSとすでにGoogle系で揃っており運用管理が一元化できる |
| 追加認証が不要 | LINE UserIDをそのままドキュメントIDとして使えるため認証基盤の追加が不要 |
| スキーマレスで柔軟 | 年表イベントの項目追加・症状記録の拡張に強い |
| 複数端末対応 | リアルタイム同期により複数端末から同じデータを参照可能 |
| 初期コスト低 | 無料枠が広く（読み取り5万回/日・書き込み2万回/日）、小規模スタートに適している |

### 6-4. Firestoreのコレクション設計（案）

```
users/{lineUserId}/
  ├── profile                        ← ユーザー基本情報（疾患名・診断日など）
  │
  ├── timeline_events/{eventId}      ← 患者年表（機能C）
  │     ├── date: "2022-04-01"
  │     ├── category: "diagnosis" | "hospitalization" | "medication_change" | "exam" | "other"
  │     ├── title: "診断確定"
  │     ├── detail: "○○病院にて潰瘍性大腸炎と診断"
  │     └── createdAt: timestamp
  │
  ├── medications/{medicationId}     ← 薬情報管理（機能A）
  │     ├── name: "ペンタサ"
  │     ├── category: "5-ASA"
  │     ├── startDate: "2022-04-01"
  │     ├── endDate: null             ← null = 現在も服用中
  │     ├── changeReason: "初回処方"
  │     ├── sideNotes: "食後に服用"
  │     └── createdAt: timestamp
  │
  └── symptom_logs/{date}            ← 症状日次記録（機能B）  ※dateをIDに使用
        ├── date: "2026-04-07"
        ├── bowelCount: 3
        ├── bristolScale: 4
        ├── bleeding: false
        ├── painScore: 2              ← 0〜5の主観スコア
        └── createdAt: timestamp
```

### 6-5. 既存よりそいとの接続方針

- 既存のCloud RunバックエンドにFirestore操作のAPIエンドポイントを追加する
- フロントエンド（LIFF）はLIFF IDを流用し、新機能用の画面を追加する
- LINE UserIDは既存の `/line/webhook` から取得済みのものをそのまま活用する
- 既存のGCS構成（診察要約・音声ファイル）は一切変更しない

### 6-6. Phase 2：診察要約との自動連携（将来設計）

現行の診察要約JSON（`summaries/{sessionId}.full.json`）には `timeline` フィールドが含まれており、将来的には診察要約から年表イベント・薬変更情報を自動抽出してFirestoreに書き込む連携が可能。

```
summaries/{sessionId}.full.json
  └── timeline フィールド → Firestore timeline_events に自動登録（Phase 2）
```

## 7. 非機能要件・設計上の注意点

| 観点 | 内容 |
|------|------|
| 入力UX | 症状記録は「1日30秒以内で完了できること」を設計の制約条件とする |
| プライバシー | 医療情報を含むため、データの保存・共有には患者の明示的な同意フローを設けること |
| 共有設計 | 情報は「患者が能動的に見せる」モデルを基本とし、病院間の自動連携は想定しない |
| LINE連携 | よりそいはLINEベースのサービスであり、各機能はLINE上のUXとして成立することを前提とする |
| 拡張性 | 機能CのタイムラインはUC以外の疾患にも転用可能な汎用設計とすること |

## 8. 用語定義

| 用語 | 定義 |
|------|------|
| 寛解期 | 潰瘍性大腸炎において炎症が抑えられ、症状が落ち着いている状態 |
| 再燃 | 寛解状態から症状が再び悪化すること |
| Bristol Scale | 便の形状を7段階で分類する国際的なスケール |
| よりそい | 株式会社メディキャンバスが提供するAI医療相談・診察サポートサービス（LINE連携） |
| 患者タイムライン | 本要件定義で定義する、患者の治療経歴を時系列で管理・表示する機能の総称 |
| LIFF | LINE Front-end Framework。LINEアプリ内でWebアプリを動作させる仕組み |
| Cloud Firestore | Googleが提供するNoSQLクラウドデータベース。本システムの患者データ保存先として採用 |
